## 1. Как всё устроено

Marketing Dronor — это автоматизированная система для холодного Twitter-аутрича. Она сама находит потенциальных клиентов, анализирует их, генерирует персональные сообщения и отправляет их от имени твоих Twitter-аккаунтов. Твоя задача как оператора — проверять сообщения перед отправкой и отвечать на входящие.

### Путь данных (полная цепочка)

```
Twitter
  ↓
M1: Profile Hunter      — находит пользователей по ключевым словам
  ↓
M2: Signal Analyzer     — присваивает tier (S/A/B/C/D) каждому профилю
  ↓
M4: Message Crafter     — генерирует персональное сообщение через Claude AI
  ↓
Очередь на проверку     — ты видишь сообщение и одобряешь или отклоняешь
  ↓
M5: Outreach Sender     — отправляет через антидетект браузер GoLogin
  ↓
M6: Inbox Monitor       — следит за ответами, генерирует follow-up
  ↓
M7: Thread Scout        — аналитика, конверсии
```

### Тиры (приоритет аудитории)

| Тир | Кто это | Фолловеры | Приоритет |
|---|---|---|---|
| **S** | Топ лидеры мнений, сильные AI/automation сигналы | 10K+ | Максимальный |
| **A** | Фаундеры, инженеры с профессиональной ролью | 1K–10K | Высокий |
| **B** | Релевантная индустрия (SaaS, tech) | 500–1K | Средний |
| **C** | Фолловеры конкурентов, упоминают конкурентов | любые | Низкий |
| **D** | Нашлись по ключевым словам, слабые сигналы | любые | Минимальный |

## 2. Первоначальная настройка

При первом входе нужно настроить API-ключи. Это делается в разделе **Admin → Settings**.

### Необходимые ключи

**Anthropic API Key** (обязательно)
- Нужен для генерации персональных сообщений через Claude AI
- Получить: https://console.anthropic.com → API Keys
- Формат: `sk-ant-api03-...`

**Twitter Bearer Token** (для M1 через официальный API)
- Нужен если используешь Twitter API v2 (не TwitterAPI.io)
- Получить: https://developer.twitter.com → Your App → Bearer Token

**TwitterAPI.io Key** (рекомендуется для масштаба)
- Альтернатива официальному API, позволяет собирать 200K+ профилей
- Получить: https://twitterapi.io
- Стоимость: ~$36 за 200K профилей
- Это основной способ работы M1 в продакшне

### Где вводить ключи

Admin панель → раздел **API Keys** → заполни поля → сохрани.

> ⚠️ Текущее ограничение: ключи сохраняются только на время сессии сервера. После перезапуска Railway нужно вводить снова. Это известная проблема — в следующем спринте будет персистентное хранение.

## 3. Добавление Twitter-аккаунтов

Раздел **Accounts** — здесь управляешь Twitter-аккаунтами, от имени которых идёт рассылка.

### Добавить один аккаунт

1. Нажать **+ Add Account**
2. Заполнить:
   - **Username** — без @, например `automation_bot_1`
   - **Status** — выбрать из списка (см. ниже)
   - **Proxy** — прокси для этого аккаунта (необязательно при добавлении)

### Массовый импорт (Bulk Add)

Кнопка **Bulk Import** — вставить список аккаунтов в формате:
```
username1
username2
username3
```
Или через запятую. Все получат статус `warming` автоматически.

### Статусы аккаунтов

| Статус | Значение | Что делать |
|---|---|---|
| `warming` | Аккаунт на прогреве, не отправляет DM | Подождать 1–3 недели |
| `active` | Полноценная работа, отправляет DM | Всё норм |
| `paused` | Временно остановлен | Вручную → active когда нужно |
| `banned` | Забанен Twitter | Не использовать |

### Стратегия прогрева (важно!)

Новые аккаунты нельзя сразу использовать для рассылки — Twitter их забанит. Прогрев:

- **Неделя 1:** только лайки и просмотры (M3 Warmup Engine делает автоматически)
- **Неделя 2:** добавить подписки
- **Неделя 3:** лёгкие комментарии
- **Неделя 4+:** можно переводить в `active` и начинать DM

Система сама управляет прогревом через **M3: Warmup Engine** — запускай его ежедневно.

## 4. Настройка прокси

Раздел **Proxies** — каждый аккаунт должен работать через отдельный прокси. Это критично для безопасности.

### Требования к прокси

- Тип: **ISP proxies** (рекомендуется IPRoyal) — не datacenter, не residential rotating
- Протокол: HTTP/HTTPS
- Один прокси = один Twitter-аккаунт (не делиться!)
- Рекомендуемый провайдер: **IPRoyal** (ipRoyal.com) — ISP proxies раздел

### Формат импорта

Нажать **Add Proxies** → вставить список в формате:
```
user:password@host:port
user:password@host:port
```

Например:
```
iproyal_user1:mypass@proxy.iproyal.com:12323
iproyal_user2:mypass@proxy.iproyal.com:12324
```

Каждая строка = один прокси. Система автоматически распарсит и сохранит.

### Проверка прокси

После импорта нажать **Check All** — система проверит каждый прокси через httpbin.org и покажет:
- `active` — работает, показывает время ответа в мс
- `dead` — не работает, показывает ошибку

Или нажать кнопку **Check** рядом с конкретным прокси.

### Назначение прокси на аккаунты

После добавления прокси перейди в раздел Accounts → отредактируй аккаунт → выбери прокси из списка. Система не даст назначить один прокси двум аккаунтам.

## 5. Data Pipeline

Раздел **Data Pipeline** — это сердце системы. Здесь запускаешь все 7 модулей по порядку.

### Правильный порядок запуска

Модули зависят друг от друга. Запускать строго по порядку:

---

### M1: Profile Hunter — сбор профилей

**Что делает:** ищет Twitter-пользователей по ключевым словам и сохраняет их в базу.

**Запуск:**
- Нажать **Run** на блоке Profile Hunter
- Или указать свой запрос в поле **Query** перед запуском

**Дефолтные поисковые запросы (встроены):**
```
("automation" OR "workflow automation") ("AI" OR "LLM") -is:retweet lang:en
("n8n" OR "zapier" OR "make.com") "automation" -is:retweet lang:en
("personal AI" OR "AI assistant") ("build" OR "building") -is:retweet lang:en
"vibe coding" ("workflow" OR "automation") -is:retweet lang:en
("solopreneur" OR "indie hacker") ("AI tools" OR "automation") -is:retweet lang:en
```

**Параметры:**
- **Query** — оставь пустым для дефолтных запросов, или введи свой Twitter-запрос
- **Max profiles** — сколько профилей собрать (дефолт 100)

**Результат:** профили появятся в счётчике `profiles_total` на дашборде.

---

### M2: Signal Analyzer — классификация по тирам

**Что делает:** берёт профили без тира, анализирует через Claude AI (bio + количество фолловеров + роль), присваивает тир S/A/B/C/D.

**Запуск:**
- Нажать **Run** на блоке Signal Analyzer
- Параметр **Batch size** — сколько профилей обработать за раз (дефолт 20)

**Стоимость:** каждый профиль = 1 запрос к Claude API. При 20 профилях = ~$0.01.

**Результат:** профили получат тир, счётчик `profiles_enriched` вырастет.

---

### M3: Warmup Engine — управление прогревом

**Что делает:** управляет аккаунтами на прогреве — назначает задачи (лайки, подписки), отслеживает прогресс, переводит аккаунты между статусами.

**Запуск:**
- Нажать **Run** → action `summary` покажет состояние всех аккаунтов
- Action `warmup` запустит задачи прогрева

**Запускать:** ежедневно для аккаунтов со статусом `warming`.

---

### M4: Message Crafter — генерация сообщений

**Что делает:** берёт профили с тирами без сообщений, генерирует персональный DM через Claude AI на основе bio и тира профиля, кладёт в очередь со статусом `pending`.

**Запуск:**
- Нажать **Run** на блоке Message Crafter
- **Batch size** — сколько сообщений сгенерировать (дефолт 20)
- **Tier filter** — оставь пустым для всех тиров, или укажи `S`, `A`, `B` и т.д.
- **Dry run** — если включить, только покажет что сгенерировало, не сохранит

**Стоимость:** каждое сообщение = 1 запрос к Claude. 20 сообщений ≈ $0.02–0.05.

**Результат:** в разделе **Queue** появятся новые сообщения на проверку.

---

### M5: Outreach Sender — отправка

**Что делает:** берёт одобренные сообщения из очереди, открывает Twitter через GoLogin (антидетект браузер), отправляет DM.

**Важно:** M5 работает только на локальном компьютере где запущен GoLogin. На Railway (облако) M5 не работает — нужен VPS или локальная машина с GoLogin.

**Требования перед запуском:**
1. GoLogin Desktop App запущен на порту 36912
2. У каждого аккаунта назначен GoLogin Profile ID
3. У каждого аккаунта назначен прокси

**Параметры:**
- **Batch size** — сколько сообщений отправить за раз (дефолт 10)

**Лимиты безопасности (ручное соблюдение сейчас):**
- Не более 50 DM в день на один аккаунт в первый месяц
- Не более 100 DM в день после прогрева
- Пауза между сообщениями минимум 3–5 минут

---

### M6: Inbox Monitor — follow-up

**Что делает:** просматривает отправленные DM, находит те на которые не ответили через N дней, генерирует warm follow-up сообщение через Claude.

**Параметры:**
- **Batch size** — сколько follow-up генерировать (дефолт 20)
- **Dry run** — проверить без сохранения

**Когда запускать:** раз в 3–5 дней.

---

### M7: Thread Scout — аналитика

**Что делает:** анализирует твиты и треды по ключевым словам для поиска возможностей ответить (не DM, а публичный reply).

**Запускать:** по необходимости, когда хочешь найти релевантные треды для участия.

---

### Рекомендуемый ежедневный порядок запуска

```
1. M1 Profile Hunter     → собрать новые профили
2. M2 Signal Analyzer    → классифицировать новые профили
3. M4 Message Crafter    → сгенерировать сообщения для S и A тиров
4. [Оператор проверяет очередь и одобряет]
5. M5 Outreach Sender    → отправить одобренные (локально, с GoLogin)
6. M3 Warmup Engine      → задачи прогрева для warming-аккаунтов
```

## 6. Очередь сообщений

Раздел **Queue** — основная ежедневная работа оператора.

### Что видишь

Каждая карточка в очереди содержит:
- **Twitter-профиль** получателя: имя, @username, bio, количество фолловеров
- **Тир** (S/A/B/C/D) с цветовой меткой
- **Сгенерированное сообщение** — персональный DM от Claude
- **Кнопки:** ✓ Одобрить / ✗ Отклонить

### Как работать с очередью

**Одобрить (✓):**
- Сообщение переходит в статус `approved`
- Оно попадёт в следующий запуск M5 и будет отправлено
- Нажимай если сообщение звучит естественно и релевантно

**Отклонить (✗):**
- Сообщение переходит в статус `rejected`
- Профиль не теряется — можно перегенерировать сообщение позже
- Нажимай если: шаблонно, странный тон, нерелевантный профиль, технические ошибки в тексте

**Одобрить всё:**
- Кнопка **Approve All** одобряет весь текущий batch
- Используй только если прочитал все сообщения и уверен в качестве

### Статусы сообщений

| Статус | Значение |
|---|---|
| `pending` | Ждёт твоего решения в очереди |
| `approved` | Одобрено, ждёт отправки через M5 |
| `rejected` | Отклонено тобой |
| `sent` | Отправлено |
| `delivered` | Twitter подтвердил доставку |
| `failed` | Ошибка при отправке |

### Советы по качеству

Хорошее сообщение:
- Упоминает что-то конкретное из bio человека
- Звучит как живой человек, не как робот
- Не начинается с "Hi! I noticed..." (шаблон)
- Короткое — 2–4 предложения максимум
- Есть конкретный крючок: почему именно этот человек

Плохое сообщение (отклоняй):
- Обобщённое, могло быть отправлено кому угодно
- Упоминает что-то чего нет в bio (галлюцинация AI)
- Слишком длинное или слишком напористое
- Грамматические ошибки

## 7. Входящие ответы

Раздел **Responses** — здесь видишь ответы на отправленные DM.

### Как работает

Система периодически проверяет входящие сообщения на всех аккаунтах и показывает треды с ответами.

### Что делать с ответом

1. Открыть тред — видишь историю переписки
2. Кнопка **Generate Reply** — Claude сгенерирует ответ на основе контекста разговора
3. Отредактировать если нужно
4. Кнопка **Send** — отправить ответ

### Приоритеты

- **Unread** — новые непрочитанные ответы, обрабатывать первыми
- Отвечать желательно в течение 24 часов — конверсия выше

## 9. Управление операторами

Раздел **Admin** — только для администратора.

### Добавить оператора

1. Admin → раздел Operators
2. Заполнить: **Имя** + **PIN** (4+ цифры)
3. Выбрать роль: `operator` или `admin`
4. Нажать Create

### Роли

| Роль | Доступ |
|---|---|
| `operator` | Очередь, Responses, Dashboard |
| `admin` | Всё + Admin панель, добавление операторов, настройки API |

### Удалить оператора

Admin → Operators → кнопка удаления рядом с именем. Нельзя удалить себя.

### Статистика операторов

Для каждого оператора видно:
- **Approved count** — сколько сообщений одобрил
- **Sent count** — сколько из них отправлено
- **Last active** — когда последний раз заходил

## 11. Частые проблемы

### "GoLogin is not running"

M5 не работает. Нужно запустить GoLogin Desktop App на компьютере где развёрнута локальная версия системы. Railway (облако) не поддерживает GoLogin.

### Очередь пустая

Значит M4 не запускался или не нашёл профилей без сообщений. Запусти M1 → M2 → M4 по порядку.

### M2 не классифицирует профили

Проверь что Anthropic API Key настроен в Admin → Settings. Без него Claude не работает.

### Profiles enriched = 0, хотя profiles total > 0

M2 ещё не запускался. Запусти Signal Analyzer.

### Прокси показывает `dead`

- Проверь правильность формата: `user:pass@host:port`
- Проверь что прокси ещё активен у провайдера
- ISP прокси иногда временно недоступны — подожди и проверь снова

### "Invalid credentials" при входе

Убедись что вводишь имя точно как создавалось (регистр важен). PIN вводи цифрами.

### Сообщения в очереди выглядят шаблонно

Это значит что профили были собраны с минимальными данными (пустой bio). M4 не имел достаточно контекста. Решение: настроить M2 на более глубокое обогащение профилей (дополнительные данные из твитов).

## Архитектура для справки

```
Railway (облако, всегда включён):
  └── Command Center (веб-интерфейс + API)
  └── PostgreSQL (база данных)

Локальный компьютер / VPS (нужен для отправки):
  └── GoLogin Desktop App (порт 36912)
  └── Marketing Dronor backend (опционально, для M5)
```

M1, M2, M3, M4, M6, M7 — работают в облаке на Railway.  
M5 (отправка через браузер) — требует GoLogin на локальной машине или VPS.
